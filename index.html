LapauApp Foundation

Project 01

Multi-screen architecture · Navigation · ViewModel · SDG 2 alignment

Problem statement

Food waste and hunger coexist in our communities — millions face hunger daily while tonnes of food go to waste. LapauApp lets users order food while automatically donating from every order to fund community meals. Every three items ordered equals one meal funded, turning everyday purchases into measurable social impact.

Lessons learned

Building a multi-screen app taught me to plan navigation upfront — sketching the flow on paper before coding saved hours of refactoring. Centralising shared data in ViewModel made the app maintainable as it grew from one screen to five. Good architecture matters more than fancy features.

LapauApp Advanced

Project 02

14 screens · Room · Firebase · Retrofit · CameraX · ML Kit

Problem statement

Extended LapauApp into a complete food waste reduction platform. Users scan food barcodes with the device camera, fetch live product details from the OpenFoodFacts API, track items in a personal fridge with expiry warnings, and donate surplus food to a real-time community food bank powered by Firebase Firestore — enabling local food sharing across the entire community.

Four technical pillars
Room Database

Local SQLite storage for fridge items and donations

Firebase Firestore

Real-time cloud sync for community sharing

Retrofit REST API

Live product lookups from OpenFoodFacts

CameraX and ML Kit

On-device barcode scanning sensor

Lessons learned

Integrating four different technologies taught me how production apps actually work. Each technology has its place — Room for privacy, Firebase for community, API for live data, sensors for input. MVVM kept everything organised despite the complexity. The hardest challenges were version compatibility, distinguishing Firestore from Realtime Database, and getting CameraX working reliably on physical devices.

Reflection

The journey, in retrospect.

From placing static text on a screen to building a 14-screen app with camera scanning, cloud sync, and real-time updates.

01

Growth across labs

Each lab built directly on the previous one — UI led to interaction, interaction led to design, design led to navigation, navigation led to persistence. By Project 2, every concept came together into one cohesive application that I could explain end-to-end.

02

Biggest challenges

  • KSP setup: matching Kotlin 2.2.10 with KSP 2.2.10-2.0.2 exactly
  • Firebase: distinguishing Firestore from Realtime Database
  • CameraX: only works on physical devices, not emulators
  • Data choices: when to use Room versus Firebase for the right data
03

SDG impact

LapauApp directly supports SDG 2 Zero Hunger. Every three items ordered funds one meal. The Community Food Bank enables real-time food sharing. My Fridge with expiry tracking prevents waste at the individual level. Small daily actions translate into measurable social impact.

04

What is next

If starting over I would invest more time in upfront planning — sketching all screens before coding. I would write unit tests from Lab 3 onwards. My next areas of growth are Compose performance optimisation and dependency injection with Hilt.
